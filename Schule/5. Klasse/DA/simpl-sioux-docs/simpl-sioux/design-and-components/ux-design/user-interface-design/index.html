    User Interface Design

    A user interface is the interface between the user (actor) and the system used by the user to achieve his goals. It shall exactly fit his needs in terms of enabling him to perform the tasks in a visual context, which is defined in the task map, and in his workflow, which can be read out from the navigation map. For this reason user interface design is based on the task cases, which are being identified during the usage-centered system analysis. The task cases focus on the real needs of the users. Without this knowledge user interface design will fail.

    NOTE:

    • Assure that system analysis had been done and that the results (descriptions of actors, their roles and their essential use cases) are available before starting to design the user interface.
    • Try to invite the same people who joined the system analysis/user story workshop before, because they already know the developed task model.

    For avoiding discussions regarding the taste of good looking user interface desgin in the early phases we recommend to follow the method of usage-centered design, step by step from abstract models to more and more realistic models.

    Navigation Map

    The navigation map shows possible user paths through application enacting task cases or scenarios. It helps to simplify the navigation within the overall architecture und organization of the user interface, especially for the focal task cases (those with highest relevance). Potential navigation problems are found very early.

    Navigation map

    • represents the essential connections/navigation paths between the clusters found in the task map,
    • answers the question "Where does the user need to go?",
    • documents interface organization to promote ease of use.

    Example

    Rules

    Only the identified task clusters (from task case map) are relevant: write each cluster name on one separate card or post-it. Then draw connection lines between these task clusters to show the navigation ways to be supported. You can differ between "roads" (thin lines for rare navigation) and "highways" (thick lines for frequent navigation).

    Abstract Prototypes

    Abstract prototypes enable developers to design the overall organization and architecture of software and web- based applications without drawing components or detailing layout. Abstract prototypes consist of an interface content model describing the contents of the various contexts within which users interact with the system plus a context navigation map showing how users move from one context to another in the course of enacting task cases. With abstract prototyping the seduction of attractive realistic prototypes can be avoided, which usually lead to weak designs (often not easily recognizable because visual design tools make even careless design look good). Furthermore, abstract models are always much simpler and stimulate the creativity of the workshop participants.

    Abstract prototypes have several advantages. First, they provide an overall picture of the user interface and show data and interactions. Second, the abstract nature of the content and navigation models allows detailed decisions to be deferred. Discussions about details are avoided.

    Figurative Prototypes

    The abstraction encourages now the creativity in design: Figurative prototypes are "Screen designs on paper" transferring the abstract components into actual controls. The experience has shown that more innovative interface designs result when an abstract prototype is used as a guide to the final visual and interaction design.

    Outgoing from the abstract prototypes, which show material and user actions on this material, the first figurative suggestions can be made. The user intention described in the task cases should lead to the appropriate UI design elements. Usually no new elements shall be invented, as there are many already defined here in the Usability Knowledge Base, a Wiki with our comprehensive knowledge, guidelines, rules and UI patterns, supports the decision making and helps to find the suitable pattern to visualize data and interactions on it.

    After performing the workshops the figurative prototypes can be refined. Details are discussed and added.

    Example

    NOTE: Figurative prototypes should fit details of the task case and work context in which task case is enacted (task map).

    For each narrative step in task cases identify abstract tools and materials

    Material

    • What information does the actor need?
    • What information does the actor need to provide to the system?
    • What information does the actor need to get from the system?
    • What events resp. conditions are triggers for this information flow?

    Tools

    • How does the actor need to operate on the material on the user interface?

    Details for the user interface design are discussed and emerged from the abstract prototypes

    Presentation design

    • What user interface pattern does fit the user intention and the material/tool identified in the abstract prototype?
    • Which layout pattern shall be used for the respective visual context?

    Interaction design

    • How can the actor operate on the interface?
    • What interaction techniques shall be provided / are expected?

    Visual Design

    At the end of this process the visual design can be made using PC-based prototyping tools. They show the exact solution and the visual design as it is expected for the final release of product/solution. These visual designs can be used in animated prototypes (i.e. Powerpoint, Visio, Expression Blend), which can be evaluated with users (customers).
