    Dialog

    Design

    Dialogs are used for property sheets, for supplementary information and high prominent messages.

    Usage

    A dialog is a type of modal pop-up that appears in front of the app content to provide critical information or ask for a decision.

    Use When

    • critical information requires a specific user task, decision, or acknowledgement
    • errors that block an apps normal operation
    • displaying detailed information or complex actions
    • the user wants to view the information in detail (remains open, even if the user changes his focus)

    Use Alternatively

    • a snackbar, when a low priority information is to be shown without prompting the user
    • a banner, when information is more prominent, but the users work does not have to be interrupted
    • a toolbar or side sheet presenting a collection of commands or tools
    • the main view with a form or table, when viewing and changing properties is the main task case
    • a wizard, when there is a complex task to be performed by the user, broken down in a series of smaller task steps
    • a drop down menu or exposed drop down menu to provide options in a non-modal, less disruptive way.

    Anatomy

    • Container - information is displayed in a rectangular, solidely colored container, extending the full width of a layout, in front of the temporarily scrimmed background (to indicate, that the dialog is modal and therefore not accessible as long as the dialog is shown)
    • Title (optional) (left aligned) - brief and concise text label in active mode describing the dialogs purpose (simple dialog), a statement (alert dialog) or a question (confirmation dialog)
    • Detail text (left aligned)
    • Actions - buttons arranged side by side at the bottom of the container: Confirming actions are placed on the right side, dismissive ones to the left of a confirming one
    • Icon - an icon (see iconography can illustrate the message optionally (i.e. warning / alarm / error).
    • For text definition see Wording

    Placement

    • Dialogs retain focus until dismissed or an action has been taken, such as choosing a setting. They shouldnt be obscured by other elements or appear partially on screen, with the exception of full-screen dialogs.
    • Usually placed in the center of the screen
    • If the click on an element has evoked the dialog, the dialog should be close to this icon and touch it at some point

    Behavior

    • As dialogs appear without warning, interrupting users work, they should be used sparingly
    • Full-screen dialogs are shown (only in Mobile scenario), when
      • components are contained which require keyboard input, such as text fields
      • when changes arent saved instantly
      • when components within the dialog open additional dialogs
    • Scrolling should be avoided. When necessary the title as well as buttons are pinned and the content in between is scrollable.
    • Dismissing a dialog is done by
      • interacting with the background
      • triggering the 'Cancel' button
    • If a close button is provided it must generally have the same effect as the 'Cancel' button at the bottom of the dialog
    • After having closed the dialog, the user is in the same context as before having opened the dialog window
