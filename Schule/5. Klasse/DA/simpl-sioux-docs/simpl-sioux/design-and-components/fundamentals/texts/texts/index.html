    Texts

    General

    Usage

    • Use a clear and simple sentence structure.
    • Use complete sentences.
    • Use a full stop for complete sentences
      • Positive example: Saves the file as a template.
      • Negative example: Save As.
    • Verbs should not be used as nouns.
    • Use present tense, if possible.
    • Avoid the passive voice when the subject is known.
    • Use the imperative in procedures to make clear who has to act.

    Correct example & incorrect example

    • All generated texts like paths shall be language independent or English.
    • Avoid redundancy. Do not give the same or very similar information to the user several times (e.g. in core text and message text).
    • User for all titles "Title Case" and otherwise "Sentence style" (refer to the corresponding tab for details).

    Exception: component or product names: first letter is always capitalized

    • Avoid phrasing that blames the user or implies user error.

    • Use gerund ("-ing" form) in a progress-indication window to show that a process is running. Use an ellipsis [ ] at the end of the sentence.

    • End with a period.

    • Do not use exclamation marks.
    • Questions have to be finished with an interrogation mark.
    • Use quotation marks around object names, except for fully qualified file paths, URLs, and domain names.
    • To indicate both singular and plural do not use parentheses t Use the plural form instead:

    Positive example: The configuration data are stored for {PLACEHOLDER} devices.

    Negative example: The configuration data are stored for {PLACEHOLDER} device(s).

    Title Case and Sentence Style

    In SiOUX according to American documentation headlines and titles in English language follow the so called Title Case while for all other user interface texts sentence style is used.
    The reason for this is that the correct capitalization ensures the readability. The capitalization of the words visually separates the headline from the other content, which helps the reader to know where an article (or a section of it) begins. Additionally the uppercased wording alerts the reader to the premise of the piece, which can be especially important when readers skim.

    Title Case

    In Title, all the major words of the title are capitalized. The following words are always lowercase in titles except when they are the first or last word: the, a, an, but, for, or, nor, to, and as. In general, do not capitalize prepositions or phrasal prepositions in titles. For example: Headline case is applied in this headline. Affordable Housing Development Stymied by Regulations.

    Usage of Title Case in Other Languages

    In German and other languages there are no extra rules for capitalization of titles. This means: Titles are capitalized exactly according to the same rules as any other German text.

    Sentence-style

    In sentence style, only the first word of the title is capitalized, and all proper names. The sentence style follows the same rules as you would use for writing a sentence.

    Wording

    Some important Dos and Donts, but never all rules

    • Avoid using unnecessary technical jargon or system-oriented information. Use short and plain words whenever possible.
    • Use everyday words and avoid words you would not say to someone else in person.
    • Be precise. Use specific verbs and choose object names and labels that clearly communicate and differentiate what the object does.
    • Keep command text short without sacrificing clarity (no more than 2-4 words, no articels).
    • For similar situations consistent words and phrasing have to be used (Double-D rule: differences are difficult).
    • Use positive phrasing. Avoid the use of negatives and do not use double negatives.
    • Describe the consequent state, not the current state (e.g. Play / Pause button in a player).
    • Avoid fillers and standard phrases.
    • Avoid noun-adjective confusion. There might be confusion between adjective and noun. You hyphenate some combined words if they form an adjective, but not if they form a noun, for example single line is a noun whereas single-line in single-line editor is an adjective.
    • Avoid using vague or generic command names (e.g. OK vs. submit)
    • Use verbs for commands that initiate an action of submit information (e.g. print, sign in).
    • Use adjectives for commands that trigger a change in the state of the system / appearance of an element (reflect how the element will change, e.g. b = bold, i = italic in Word)
    • Include ellipses ("...") in command text to indicate when more information will be required (e.g. open ...)
    • Avoid using the camel and pascal notation to abbreviate terms (e.g. use "Option button" instead of "optionButton" or "OptionButton").
    • Avoid abbreviations: Abbreviations are often internally known but unknown to the user. Therefore, avoid abbreviations: If the space is limited, abbreviate the terms. Abbreviate the terms the way that the meaning is apparent. This means a word should only be abbreviated as much as necessary. Use tooltips to provide the long form of the abbreviation.
    • Abbreviate units of measurement when they are accompanied by a numeric value. Always place one space between a number and a unit.

    Phrasing Templates

    • Avoid camel notation/pascal notation to abbreviate terms.

    Correct/incorrect Example

    • When possible, state messages, that require a choice as questions that permit "Yes" or "No" as an answer.

    Correct example

    • If a problem occurs try to state the problem, its cause (if possible) and offer direct support.
    • A problem may have one out of several possible reasons. In this case state the possible reasons, connected by "or". Allocate an individual line for each reason and each "or". Avoid mapping more than three reasons in one message.
    • Keep the view of the customer in mind: Compose messages as specific as possible. E.g. in message boxes the user isn't interested in a description of the technical reason for the problem but in a solution.

    Correct/incorrect Example

    • The information given should be as short, though complete as possible. Give only necessary background information. Especially in message box message texts should not exceed two to three lines. If there are still details missing provide a help button that opens a help window.
